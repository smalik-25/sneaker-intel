# Choosing real data over a flaky scraper

The original plan had three live sources: eBay sold listings, Reddit, and Google Trends. Partway through I hit the question every data project hits eventually. Where does the real data actually come from? The answer reshaped the project.

The constraints were honest ones. eBay and Reddit both need API keys and approval. And the tempting shortcut, a community scraper that pulls StockX, GOAT, Flight Club, and Stadium Goods prices, turned out to be exactly the thing you don't want load-bearing. It scrapes sites sitting behind anti-bot protection, it has open issues about 403s when those defenses tighten, and it's Node where the rest of my project is Python. Fine for a hobby tracker. A bad foundation for something with my name on it that I want to demo reliably.

So I split the difference. The real backbone is the Kaggle StockX 2019 dataset: roughly 99,000 real Off-White and Yeezy resale sales, with sale price, retail price, release date, size, and buyer region. No keys, no scraping, no terms-of-service gray zone, and it won't break the morning before an interview. It also maps almost one-to-one onto the schema I'd already built. Sales flow into `fact_sales`, and the loader derives `dim_drops`, retail price and release date, straight from the data instead of me hand-seeding it. The Off-White and Yeezy concentration is honestly ideal for studying premium behavior. Those are the two engines of the 2017 to 2019 hype cycle, deadstock that traded at multiples of retail, and the dataset captures that whole arc. I kept Google Trends too, since it's a genuinely live, key-free demand signal.

What I thought hardest about was what not to throw away. I'd already written the eBay and Reddit clients, real API integration, the kind of thing worth showing. Deleting them would trade a genuine skill demonstration for a slightly tidier repo. So I reframed them as documented, tested, schema-ready future extensions. The loader and warehouse already support them; turning them on later is additive. The framing becomes "multi-source ingestion with live API clients plus a real historical dataset," which is more honest and more mature than either alone.

One detail that mattered for correctness: StockX names are slugs like `Adidas-Yeezy-Boost-350-V2-Zebra`. I normalize those into clean text and use that as the single conformed key in `dim_shoes`, so sales and the live Trends signal land on the same shoe instead of two near-duplicates that never join. Small step, but it's the difference between a star schema that works and one that quietly doesn't.

The takeaway I'm keeping: real data doesn't have to mean a live API. A solid public dataset, wired in cleanly, beats a fragile feed for a project whose whole point is the engineering around the data.
